# 家教在线平台 - 前提假设与约束条件

## 1. 概述

本文档记录家教在线平台改造过程中的所有前提假设、设计约束和技术决策，确保团队达成共识。

---

## 2. 支付与商业模式

### 假设 A1：本期不接入支付系统
- VIP 会员由管理员在后台手动开通
- 不涉及在线支付、退款等流程
- 教员与学员之间的费用线下结算
- 后续版本可对接微信支付/支付宝

### 假设 A2：VIP 权益定义
- VIP 教员：每日申请次数无限制、资料排序靠前、可查看学员联系方式
- VIP 学员：需求可置顶、查看教员联系方式
- 非 VIP 用户部分功能受限（如每日申请上限5次）

---

## 3. 短信服务

### 假设 B1：开发环境使用 Mock 模式
- 复用 roncoo-education 现有 SMS 模块
- 开发和测试环境短信验证码统一为 `123456`
- 短信发送记录正常写入数据库，但不实际调用短信网关
- 生产环境切换为真实短信网关，通过配置项 `sms.mock.enabled=true/false` 控制

### 假设 B2：短信使用场景
- 用户注册验证码
- 用户登录验证码
- 预约确认通知（后续版本）

---

## 4. 网关与路由

### 假设 C1：复用现有 Gateway 路由体系
- 所有家教相关接口均挂载在 user 服务下
- 路由前缀保持 `/user/{api|auth|admin}/` 三层结构
- 无需新增微服务，在 `roncoo-education-user` 模块中开发

### 假设 C2：鉴权机制
- JWT Token 机制不变
- Token 中新增 `userType` 字段标识用户角色
- Gateway 层面根据路径前缀做基础鉴权
- 业务层面在 Controller 中做细粒度角色校验

---

## 5. 代码架构

### 假设 D1：遵循 MyBatis 现有模式
- 使用 MyBatis-Plus 作为 ORM 框架
- Mapper XML 文件放在 `resources/mapper/` 目录下
- Service 层使用 `IService` / `ServiceImpl` 模式
- 分页使用 MyBatis-Plus 的 `Page` 对象

### 假设 D2：项目包结构
```
com.roncoo.education.user
├── dao
│   ├── impl          # DAO 实现
│   └── mapper        # Mapper 接口
├── service
│   ├── api           # Public API Service
│   ├── auth          # Auth API Service
│   └── admin         # Admin API Service
├── controller
│   ├── api           # Public API Controller
│   ├── auth          # Auth API Controller
│   └── admin         # Admin API Controller
└── common
    ├── bo            # Business Object
    ├── dto           # Data Transfer Object
    └── vo            # View Object
```

### 假设 D3：代码生成
- 使用 MyBatis-Plus Generator 生成基础代码
- 生成后手动调整业务逻辑
- Entity 类使用 Lombok 注解

---

## 6. 数据与存储

### 假设 E1：数据库
- 使用现有 MySQL 实例，同库新建表
- 表名前缀不做区分，通过表名语义区分业务
- 字符集 UTF8MB4，引擎 InnoDB

### 假设 E2：文件存储
- 教员认证材料（身份证照片、学历证等）复用现有文件上传模块
- 支持本地存储和 OSS 存储（通过配置切换）
- 文件大小限制 5MB
- 支持格式：JPG、PNG、PDF

### 假设 E3：缓存
- 热点数据使用 Redis 缓存
- 字典数据全量缓存，TTL 24小时
- 教员列表页缓存，TTL 5分钟
- 用户 VIP 状态缓存，TTL 1小时

---

## 7. 业务规则

### 假设 F1：用户角色
- 一个用户可以同时是教员和学员（userType=3）
- 注册时选择角色，后续可在个人中心切换/新增角色
- 管理员角色独立，不可兼任教员/学员

### 假设 F2：教员认证
- 认证为非强制流程，但未认证教员在列表中排名靠后
- 认证审核由管理员在后台手动完成
- 审核不通过需填写拒绝原因
- 修改关键信息（姓名、学历、院校）后需重新认证

### 假设 F3：需求发布
- 需求发布后30天自动过期
- 过期后状态变更为"已关闭"
- 用户可手动关闭或标记为"已完成"
- 每个用户同时最多5个有效需求

### 假设 F4：申请限制
- 非 VIP 教员每天最多申请5个需求
- VIP 教员不受此限制
- 同一教员对同一需求只能申请一次

### 假设 F5：预约规则
- 教员需在24小时内确认预约
- 超时未确认自动取消
- 确认后2小时内不可取消
- 一个教员同一时段最多一个确认中的预约

---

## 8. 非功能性假设

### 假设 G1：性能
- 目标并发用户数：500
- 页面响应时间：< 2秒
- 接口响应时间：< 500ms

### 假设 G2：安全
- 敏感信息（手机号、真实姓名）部分脱敏显示
- 非 VIP 用户不可查看完整联系方式
- 上传文件进行类型和大小校验
- 接口限流：同一 IP 每分钟最多60次请求

### 假设 G3：兼容性
- 门户网站支持主流浏览器（Chrome、Firefox、Safari、Edge）
- 移动端通过响应式布局适配（不单独开发APP）
- 管理后台仅支持桌面浏览器
